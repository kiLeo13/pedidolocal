Pedido Local API - Documentação Técnica

O Pedido Local é uma API REST monolítica para venda e entrega de produtos dentro de uma cidade configurada. O foco do produto é permitir que clientes comprem bebidas, sucos, lanches, sobremesas, vegetais e outros itens gerais, com controle de catálogo, pedidos, estoque, pagamento registrado, autorização por papéis e trilha de auditoria.

Esta documentação descreve a implementação atual do backend. Não há frontend, serviços externos, cloud, mensageria ou integração real com meios de pagamento.

Conceitos Essenciais Primeiro

Como os IDs funcionam

  • Todos os principais registros usam id inteiro positivo gerado pelo banco SQLite.
  • IDs são chaves primárias autoincrementais e aparecem nas respostas para referência entre entidades.
  • Campos que recebem IDs pela API, como category_id e product_id, aceitam valores entre 1 e 2147483647.
  • Quando um pedido é criado, os itens gravam também um snapshot do nome e preço do produto, então alterações futuras no catálogo não reescrevem o histórico do pedido.

Como timestamps são tratados

  • Campos created_at e updated_at são preenchidos pela aplicação no padrão UTC.
  • Entidades com ciclo de pedido podem ter canceled_at e delivered_at, também em UTC.
  • O SQLite não armazena timezone de forma tão rica quanto bancos como PostgreSQL; por isso a aplicação sempre gera os horários com datetime.now(UTC).
  • O objetivo é manter consistência técnica e facilitar auditoria de eventos.

Como a autorização funciona

  • A API usa JWT Bearer Token.
  • Clientes se registram em POST /auth/register e fazem login em POST /auth/token.
  • Admins não são criados por endpoint público; são criados pelo comando local pedido-local create-admin.
  • Rotas públicas permitem leitura de catálogo e health check.
  • Rotas de cliente exigem usuário autenticado com papel customer.
  • Rotas administrativas exigem papel admin.
  • Tentativas sem token, com token inválido ou com papel insuficiente geram logs de auditoria quando aplicável.

Validações e Limitações Gerais

  • Corpo HTTP acima de 1 MiB é rejeitado com 413.
  • Textos são aparados com strip() antes de validar tamanho.
  • Textos obrigatórios não podem ficar vazios depois do trim.
  • Campos de nome, endereço e cidade não aceitam caracteres de controle.
  • Descrições aceitam quebras de linha, tabulação e retorno de carro, mas rejeitam outros caracteres de controle.
  • Preços aceitam no máximo 99999.99, com duas casas decimais.
  • Estoque aceita valores entre 0 e 100000.
  • Quantidade por item de pedido aceita valores entre 1 e 100.
  • Um pedido pode conter entre 1 e 50 itens.
  • Produtos duplicados no mesmo pedido são rejeitados.
  • Senha de cadastro exige 8 a 128 caracteres, letra minúscula, letra maiúscula e número.
  • Telefone aceita apenas dígitos, espaços, +, - e parênteses, com 8 a 15 dígitos reais.
  • Data de nascimento deve estar entre 1900-01-01 e a data atual.

Autenticação

O login usa o padrão OAuth2 Password Flow do FastAPI. O campo username deve receber o e-mail do usuário. O retorno contém access_token e token_type.

Exemplo de uso

POST /auth/token
Content-Type: application/x-www-form-urlencoded

username=cliente@example.com&password=Customer12345

Depois do login, envie o token em rotas protegidas:

Authorization: Bearer <access_token>

Regras de Processo

  • Criar pedido valida cidade, produtos ativos, categoria ativa, estoque e restrição de idade.
  • O estoque é reduzido na mesma transação da criação do pedido.
  • Se uma validação falha, a operação registra rejeição auditável e não altera estoque.
  • Pedido entregue não pode ser cancelado.
  • Cancelamento restaura estoque uma vez e marca stock_restored.
  • Status de pedido só segue transições documentadas.
  • Status de pagamento só segue transições documentadas.

Transições de pedido

pending -> confirmed | canceled
confirmed -> preparing | canceled
preparing -> out_for_delivery | canceled
out_for_delivery -> delivered
delivered -> estado final
canceled -> estado final

Transições de pagamento

pending -> paid | failed
failed -> pending | paid
paid -> refunded
refunded -> estado final

Auditoria e Segurança

  • Eventos críticos geram registros em audit_logs.
  • São auditados cadastro, sucesso/falha de login, negação de autorização, mutações de catálogo, criação/cancelamento de pedido, mudança de status e pagamento.
  • Senhas, hashes, JWTs e tokens nunca devem aparecer em respostas ou logs.
  • Autorização é aplicada por dependências FastAPI centralizadas.
  • Persistência usa SQLAlchemy, evitando concatenação manual de SQL.
  • Validações de entrada reduzem risco de abuso de tamanho, payload inválido e manipulação de dados.

Códigos de Resposta e Erros

CódigoUso típico
200Leitura ou mutação concluída com sucesso.
201Criação de usuário, categoria, produto ou pedido.
400Regra de negócio inválida, como cidade fora da área.
401Token ausente, inválido ou credenciais incorretas.
403Papel insuficiente ou restrição de idade.
404Recurso inexistente, inativo ou invisível para o usuário.
409Conflito, como e-mail duplicado, estoque insuficiente ou transição inválida.
413Corpo da requisição maior que o limite configurado.
422Payload falhou nas validações Pydantic.

Erros de domínio retornam o formato simples:

{ "detail": "mensagem do erro" }

Testes e Validação Técnica

Os testes usam bancos SQLite temporários migrados pelo Alembic, não mocks de persistência. Isso valida o contrato API + service layer + ORM + banco. 

.\.venv\Scripts\python -m pytest --cov=app --cov-report=term-missing
.\.venv\Scripts\python -m ruff check .
.\.venv\Scripts\python -m alembic upgrade head
  • tests/test_auth.py: cadastro, login, duplicidade, proteção de rota.
  • tests/test_catalog.py: permissões admin, leitura pública, soft delete.
  • tests/test_orders.py: criação, estoque, snapshots, cancelamento, isolamento e transições.
  • tests/test_audit.py: criação de logs e ausência de segredos.
  • tests/test_validation.py: limites de payload, senha, telefone, datas, produto, preço e estoque.

Limitações Conhecidas

  • Sem pagamento real; apenas registro de método e status.
  • Sem cálculo de frete, rotas ou entregadores.
  • Sem UI.
  • Sem serviços externos.
  • SQLite é adequado para o escopo acadêmico/local, mas uma produção real normalmente exigiria PostgreSQL ou similar.
